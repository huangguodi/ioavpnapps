# 性能优化清单

## 说明

- 本清单仅整理性能优化方向，不修改现有业务功能、不改变现有交互语义、不调整接口协议。
- 所有优化以“不影响原功能逻辑、禁止引入新 Bug”为前提。
- 本次明确排除：背景图资源压缩、背景图分辨率策略、背景图切换方案。
- 优化原则：
  - 先做低风险、可回退、可验证的优化。
  - 先减少轮询、日志、平台通道、磁盘 I/O，再处理次级体验问题。
  - 先做轻量接口替代全量状态读取，避免无意义的大对象解析。

## 总体优化目标

- 降低 Flutter 主线程被频繁打断的概率。
- 降低 Android / iOS / Windows 平台通道往返次数。
- 降低运行期间的定时任务叠加开销。
- 降低 SharedPreferences / 文件系统频繁写入带来的卡顿。
- 降低节点相关全量数据读取、解析、测速带来的卡顿。
- 启动流程按“必须首屏完成”和“可延后后台完成”拆分。

## 全平台通用优化项

### P0

- [已完成] 降低用户信息轮询成功后的本地落盘频率。
  - 当前 `fetchUserInfo()` 成功后会写入 `SharedPreferences`。
  - 优化方向：仅在关键字段变化时写盘，或改为页面退出 / 切后台 / 显式刷新后再写盘。
  - 前提：不改变当前登录态和缓存恢复逻辑。
  - 完成情况：已改为仅在缓存内容变化时写盘，并同步复用到登录、本地缓存恢复、礼品卡成功后的用户信息持久化流程。

- [已完成] 减少多条定时任务长期并行运行。
  - 当前长期任务包括：
    - 流量刷新
    - 用户信息刷新
    - Mihomo 守护检查
    - 安全检查
  - 优化方向：统一调度频率，非活跃状态下降频，页面不可见时暂停非关键轮询。
  - 前提：不影响 VPN 运行监控和异常恢复逻辑。
  - 完成情况：用户信息刷新、安全检查、Mihomo 守护检查均已改为单次调度续期，避免固定 `Timer.periodic` 在任务耗时抖动时叠加；同时将首轮执行时间错开，降低 5 秒周期任务同时命中主线程与平台通道的概率。

- [已完成] 将“查询单一状态”从“读取整份数据”中拆出来。
  - 适用字段：
    - 当前模式
    - 当前选中节点
    - 是否运行中
  - 优化方向：平台侧提供轻量接口，不再为了一个字段读取整份 `proxies` 数据。
  - 前提：保留原有全量接口用于完整节点弹窗或完整状态页。
  - 完成情况：已补齐 iOS 当前节点轻量查询链路，三端当前节点查询均不再依赖 Flutter 侧先拉整份 `proxies` 再解析。

### P1

- [已完成] 启动流程拆分为“首屏必需”和“后台补充”两阶段。
  - 首屏必需：
    - 基础登录态判断
    - 必要的 Mihomo 启动判断
    - 基础页面可展示状态
  - 后台补充：
    - 非关键刷新
    - 非关键缓存准备
    - 可延后节点详情同步
  - 前提：不改变热更新、登录和安全策略的执行结果。
  - 完成情况：启动阶段已只保留热更新、登录态判断、登录、礼品卡校验、Mihomo 启动和初始模式解析；Android 通知/电池优化权限申请已改为首页跳转后后台触发，不再阻塞首屏进入。

- [已完成] 节点相关逻辑从“全量刷新”改为“按需刷新”。
  - 节点列表打开时先使用缓存展示。
  - 当前节点变化只更新选中项，不强制重拉全量详情。
  - 节点测速采用分批或并发池，避免一次性串行全部测速。
  - 前提：不改变当前节点选择结果和测速结果展示规则。
  - 完成情况：节点面板已改为优先展示缓存并后台补拉；节点切换成功后仅刷新必要字段；节点测速已改为小批量并发分批回填，节点相关逻辑已从全量刷新收敛为按需刷新。

### P2

- [已完成] 弹窗、广告轮播、扫码等次级模块降低后台活跃度。
  - 非显示状态下停止定时器。
  - 网络图加载限制尺寸与更新频率。
  - 避免弹窗内重复触发高频刷新。
  - 前提：不改变现有功能入口和交互路径。
  - 完成情况：广告轮播已支持交互冷却、悬停暂停和应用非活跃暂停；工单弹窗轮询已改为单次调度续期并在应用不可见时暂停；扫码绑定弹窗在应用切后台时会停止相机扫描，恢复前台后再继续。

## Android 优化清单

### P0

- [已完成] 关闭或收敛高频 Mihomo 原生日志向 Flutter 回传。
  - 问题：
    - Android 侧日志通过事件通道实时推送。
    - 日志回传中包含 `runOnUiThread`。
    - 高频连接日志会持续打断 UI 线程。
  - 优化方向：
    - 默认关闭高频连接类日志。
    - 对日志做采样、聚合或批量回传。
    - 将仅调试需要的详细日志放到显式调试开关下。
  - 风险控制：
    - 保留错误日志、崩溃日志、关键状态日志。
    - Release 默认收敛，Debug 保持可追踪。
  - 完成情况：已将高频连接日志默认切为收敛模式，并避免重复订阅日志流。

- [已完成] 将流量采样逻辑从主线程 Looper 迁移到后台线程。
  - 问题：
    - 当前流量事件由 `Handler(mainLooper)` 每秒执行。
    - 底层流量读取一旦抖动，会直接影响主线程。
  - 优化方向：
    - 后台线程读取流量数据。
    - 主线程仅做轻量事件分发。
    - 对短时间无变化数据做节流推送。
  - 风险控制：
    - 不改变流量展示值。
    - 不改变前台服务和 Mihomo 实际运行逻辑。
  - 完成情况：已将 Android 侧流量读取迁移到后台 `HandlerThread`，主线程仅负责回传事件。

- [已完成] 减少完整代理数据 `getProxies()` 调用次数。
  - 问题：
    - 节点状态刷新、节点弹窗、节点切换后校验都可能依赖完整代理数据。
    - 大 JSON 读取和 Dart 解析会带来卡顿尖峰。
  - 优化方向：
    - 增加轻量接口：
      - 获取当前选中节点
      - 获取当前模式
      - 获取节点简表
    - 全量代理数据仅在真正需要完整节点面板时拉取。
  - 风险控制：
    - 保留现有 `getProxies()` 作为兜底接口。
  - 完成情况：Android 已补齐当前模式、当前选中节点、当前节点详情等轻量查询链路；首页节点状态刷新优先走轻量接口，仅在真正需要完整节点面板时再读取整份 `proxies`，并结合 Flutter 侧同类请求合并减少重复全量读取。

### P1

- [已完成] 收敛 Android 运行期轮询叠加。
  - 当前叠加任务：
    - 流量 1 秒
    - 用户信息 5 秒
    - 守护检查 5 秒
    - 安全检查 5 秒
  - 优化方向：
    - 合并调度时机。
    - 页面不可见或窗口非活跃时降频。
    - Mihomo 未运行时减少无意义轮询。
  - 风险控制：
    - 守护恢复能力不下降。
    - 安全检查仍保留但改为更合理频率。
  - 完成情况：用户信息轮询延续既有单次调度；安全检查和守护检查已同步改为单次调度续期并增加首轮错峰，避免 Android 侧多个 5 秒任务在同一时刻集中执行。

- [已完成] 降低 `fetchUserInfo()` 高频写盘。
  - 优化方向：
    - 仅在 `quota`、`expire_time`、`ads`、登录态相关字段变化时更新缓存。
    - 避免每次轮询成功都写盘。
  - 风险控制：
    - 保证崩溃重启后仍能恢复必要用户信息。
  - 完成情况：`fetchUserInfo()` 已统一复用按变化写盘逻辑，Android 与其他平台一致，仅在缓存内容实际变化时才更新本地 `SharedPreferences`；Windows 额外保留延迟批量落盘策略，避免高频轮询期间反复写盘。

- [已完成] 节点测速改为并发池或分批模式。
  - 问题：
    - 当前“全部测速”是逐个串行执行。
    - 节点较多时页面长时间处于频繁状态更新中。
  - 优化方向：
    - 限制并发数，例如 3~5 个一组。
    - 分批回填 UI。
    - 首次只测当前节点或首屏可见节点。
  - 风险控制：
    - 不改变节点切换结果。
    - 不改变最终测速数值展示规则。
  - 完成情况：已改为小批量并发测速并分批回填 UI，避免弹窗内长时间逐条串行刷新。

### P2

- [已完成] 启动阶段延后非关键任务。
  - 优化方向：
    - 权限申请在合适时机再触发。
    - 首页可见后再做非关键资源准备。
    - 节点详情和广告信息放到首屏后补拉。
  - 风险控制：
    - 不影响热更新优先级。
    - 不影响登录态校验和启动成功率。
  - 完成情况：Android 启动权限申请已从 Splash 阶段移到首页跳转后的后台任务；首页图片预缓存继续由 `HomePage` 在进入页面后异步完成，避免 Splash 额外等待。

## iOS 优化清单

### P0

- [已完成] 缓存 `NETunnelProviderManager`，避免每次调用都 `loadAllFromPreferences`。
  - 问题：
    - iOS 平台命令发送前会先加载 TunnelManager。
    - 该过程系统开销明显，高频调用时容易造成卡顿。
  - 优化方向：
    - App 启动后缓存可用的 `NETunnelProviderManager`。
    - 状态未变时复用缓存对象。
    - 仅在必要时重新同步 Preferences。
  - 风险控制：
    - 增加缓存失效兜底。
    - manager 异常时自动回退到原加载流程。
  - 完成情况：已增加启动预热、缓存复用、并发加载合并和保存成功后的缓存回填。

- [已完成] 减少 `sendProviderMessage` 高频往返。
  - 问题：
    - iOS Extension 通讯成本高于普通方法调用。
    - 流量、模式、节点、代理等细碎请求叠加容易卡。
  - 优化方向：
    - 合并多个小请求。
    - 降低无变化时的重复请求。
    - 将常用状态合并成单次轻量返回。
  - 风险控制：
    - 不改变现有 Tunnel 建立、停止、切换流程。
  - 完成情况：已为 `isRunning`、当前模式、当前选中节点等轻量状态增加短 TTL 内存缓存，并在启动、停止、模式切换、节点切换、代理刷新后主动回填或失效，减少重复的小请求往返，尤其降低 iOS Extension 高频状态查询压力。

- [已完成] 为“当前模式 / 当前节点 / 是否运行”提供轻量查询接口。
  - 问题：
    - 当前有些逻辑会为了拿当前节点重新获取整份 `proxies`。
  - 优化方向：
    - provider 直接返回简单字符串或布尔值。
    - 避免整份 JSON 来回传输和解析。
  - 风险控制：
    - 保留完整接口给节点列表页。
  - 完成情况：iOS Provider 已提供当前模式、当前节点、运行状态等轻量查询，Flutter 侧接入后优先走轻量链路；三端当前节点详情刷新均已不再默认依赖整份 `proxies`。

### P1

- [已完成] 下调 iOS 流量刷新频率或做动态频率控制。
  - 优化方向：
    - 页面不可见时降频。
    - 速率长期无变化时降频。
    - 刚进入首页时短时高频，稳定后低频。
  - 风险控制：
    - 不改变用户看到的连接状态。
    - 不影响异常状态识别。
  - 完成情况：iOS 流量事件已改为自适应单次调度，应用非活跃时自动降到低频，速率与累计流量连续稳定时降低拉取频率，并对无变化数据减少重复回传，保留活跃阶段的快速刷新体验。

- [已完成] 首页初始化减少同步向 Extension 拉状态。
  - 问题：
    - 首页一进来若同时查模式、查节点、查完整代理，会明显拖慢交互。
  - 优化方向：
    - 优先展示基础状态。
    - 节点详情稍后补充。
    - 当前节点使用轻量接口优先返回。
  - 风险控制：
    - 页面最终展示数据保持一致。
  - 完成情况：首页初始化已不再并行同步拉模式与节点详情；基础运行状态和模式优先返回，当前节点详情改为在基础状态确定后延后补充，并复用轻量状态缓存减少 iOS Extension 重复查询。

- [已完成] 减少 JSON 序列化 / 反序列化。
  - 优化方向：
    - 能返回字符串的不返回大 JSON。
    - 能返回数值的不包装复杂结构。
    - 常用字段走专用通道。
  - 风险控制：
    - 不变更业务字段含义。
  - 完成情况：iOS Runner 与 PacketTunnel 针对 `getMode`、`getProxies`、`getSelectedProxy`、`urlTest` 改为轻量字符串消息直传，避免这几条高频链路继续用 JSON 包装字符串再做额外序列化 / 反序列化。

### P2

- [已完成] Tunnel 启动后的状态等待和附加查询节流。
  - 优化方向：
    - 避免刚启动成功后马上发起多次额外查询。
    - 将非关键确认动作稍后执行。
  - 风险控制：
    - 不影响 Tunnel 连接成功判断。
  - 完成情况：iOS 在 Tunnel 连接成功后新增短暂稳定窗口，`getMode`、`getSelectedProxy`、`getProxies` 等附加状态查询会自动延后到稳定窗口后再执行；Flutter 侧同时延后非关键节点状态补拉，减少启动成功瞬间对 Extension 的重复查询。

## Windows 优化清单

### P0

- [已完成] 减少全量代理数据读取与解析。
  - 问题：
    - Windows 虽已有部分轻量优化，但很多逻辑本质上仍依赖整份 `proxies` 数据。
  - 优化方向：
    - 原生层直接提供：
      - 当前选中节点
      - 当前模式
      - 节点简表
    - 节点详情页再取全量数据。
  - 风险控制：
    - 保留原有全量接口兜底。
  - 完成情况：Windows 已提供当前选中节点、当前模式、节点简表等轻量链路；当前节点变化时优先走原生轻量详情返回，不再为了首页当前节点展示回退到整份 `proxies` 读取，完整节点详情仅在真正需要时再取。

- [已完成] 降低 5 秒轮询后的频繁写盘。
  - 问题：
    - Windows 小窗对频繁 I/O 抖动感知明显。
  - 优化方向：
    - 用户信息缓存按变化写盘。
    - 非关键字段不立即落盘。
  - 风险控制：
    - 保证异常退出后仍能恢复核心信息。
  - 完成情况：Windows 下用户信息 5 秒轮询已改为按变化延迟批量落盘，轮询期间最多按节流间隔写盘一次；登录、礼品卡成功、清理登录态等关键路径仍保持立即持久化，退出前会主动补刷待写缓存。

### P1

- [已完成] 收敛守护检查与状态查询重叠。
  - 问题：
    - 守护检查、用户信息刷新、节点信息刷新在同一时段叠加。
  - 优化方向：
    - 统一调度。
    - 避免多个 5 秒任务同时执行。
    - 将状态检查错峰执行。
  - 风险控制：
    - Mihomo 异常恢复能力不降低。
  - 完成情况：Windows 守护检查执行时会为非关键状态查询预留短暂冷却窗口，首页节点状态刷新会自动避开该时间片；同时轻量状态与代理查询增加同类请求合并，减少守护检查与状态读取在同一时刻重复穿透平台通道。

- [已完成] 节点弹窗改为“先秒开，再补充数据”。
  - 优化方向：
    - 先显示缓存列表。
    - 当前选中节点即时展示。
    - 延迟信息按需填充。
    - 全部测速不在弹窗打开瞬间执行。
  - 风险控制：
    - 不改变节点切换逻辑。
    - 不改变节点排序最终结果。
  - 完成情况：已优先使用缓存节点列表秒开弹窗，存在缓存时在弹窗展示后后台刷新节点数据，并保留手动刷新入口。

- [已完成] 广告轮播和网络图降低对小窗交互的干扰。
  - 优化方向：
    - 非必要时暂停自动轮播。
    - 限制网络图刷新和解码触发频率。
    - 打开弹窗或交互时暂缓轮播。
  - 风险控制：
    - 不影响广告点击跳转功能。
  - 完成情况：已为广告轮播增加交互冷却、悬停暂停、应用非活跃暂停，并降低网络图渲染质量以减少小窗交互干扰。

### P2

- [已完成] 在 Windows 小窗场景进一步减少无意义刷新。
  - 优化方向：
    - 速度数值未变化不刷新。
    - 节点状态未变化不刷新。
    - 广告数据未变化不刷新。
  - 风险控制：
    - 页面显示值保持真实有效。

## 最终原则

- 优先做低风险减负，不做大规模重构。
