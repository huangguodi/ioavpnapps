一、Flutter 层（主 App）坑 + 优化
真实坑点
Flutter 不能直接访问 NetworkExtensionFlutter 运行在主 App 进程，TUN /fd/ Mihomo 全在拓展进程，两个进程完全隔离。→ 不能直接通信、不能直接传值、不能直接控制。
无法用 Flutter 代码启动 / 停止 VPN必须通过 iOS 原生 API NETunnelProviderManager 间接启动。
Flutter UI 容易显示错误状态拓展被杀、VPN 断连、启动超时，Flutter 收不到实时回调。
内存爆炸会拖死拓展Flutter 图片 / 列表 / 桥接内存占用太高，会导致系统优先杀低优先级的 VPN 拓展。
必须优化
使用 Channel + 共享内存 / AppGroup 做进程通信
Flutter 只做：配置写入、启动指令、状态展示
禁止在 Flutter 里做任何网络操作
图片压缩、懒加载、减少内存占用
用 UserDefaults(suiteName: AppGroup) 同步配置
1. 主 App 与 NetworkExtension 是两个完全独立进程
Flutter 无法直接控制拓展
无法直接传递 fd、无法直接启停 Mihomo
无法直接读日志、状态
→ 必须用 IPC 通信
2. Flutter 侧无法监听 VPN 系统级状态
系统断 VPN、拓展崩溃、被系统 kill
Flutter 收不到回调，UI 状态永远不对
→ 必须靠 Darwin Notification + AppGroup 同步
3. 配置必须双进程读写，Flutter 写入后拓展才能读
直接写 SharedPreferences 拓展看不见
必须存在 AppGroup 共享目录
多进程并发读写会配置损坏
4. Flutter 内存占用大会导致拓展优先被杀死
iOS 内存紧张时，系统优先杀 Extension
Flutter 图片、列表、桥接缓存大 → 拓展秒死
→ 成熟 App 都会限制主 App 内存
5. Flutter 无法直接调用 NE 权限 API
必须通过 MethodChannel 转给 iOS 原生执行。
注意：不能影响win android平台！

二、iOS 原生层（Runner / 主 App 原生）坑 + 优化
真实坑点
主进程完全拿不到 TUN fd任何取 fd 代码写在主 iOS 原生里 一定返回 nil。
NETunnelProviderManager 启动延迟、偶发失败系统 API 不稳定，特别是 iOS 17+、iOS 26。
主进程和拓展进程数据不同步配置、密钥、节点列表、规则必须双进程读同一份文件。
VPN 权限必须用户手动授权授权失败不会抛异常，只会静默失败。
必须优化
原生只做：VPN 权限申请、启动 / 停止拓展、状态回调
配置统一存入 AppGroup 共享目录
做启动重试机制（最多 3 次）
监听系统 VPN 状态，实时同步给 Flutter
1. 主进程绝对拿不到 TUN fd
任何取 fd 代码写在这里一律返回 nil
fd 只属于拓展进程
2. VPN 配置只能用 NETunnelProviderManager 管理
不能直接创建 TUN
必须安装系统 VPN 配置
授权弹窗必须用户点允许
授权失败静默失败，不抛异常
3. 主进程不能读写 TUN、不能包转发
所有流量处理完全不能在主进程做
4. 必须处理系统 VPN 状态同步
手动在系统设置关闭 VPN
拓展崩溃
系统回收拓展
主原生都要监听并同步回 Flutter。
5. 拓展崩溃主进程不知道
必须用：
Darwin Notify
共享文件标记
重启监听机制
否则 UI 永远显示 “已连接”。

三、NetworkExtension 拓展层（最核心、坑最多、决定死活）
真实坑点（全是致命坑）
内存硬上限 20~25MB（iOS 16~26）超了直接被系统 kill，无日志、无报错。
拓展启动超时 25 秒超过系统直接杀死，Mihomo 还没启动完就死。
TUN fd 必须在拓展内部获取主进程拿不到，Flutter 更拿不到。
单线程读写 TUN，不能并发多线程读写 → 直接内核报错、闪退、断流。
拓展后台极易被系统杀死锁屏、无流量、高流量、其他 App 耗内存都会被杀。
不能用任何异步延时太长的操作拓展不支持常规后台异步。
iOS 16+ 不能使用官方 readPackets/writePackets用了就和 fd 冲突，必崩。
必须优化
取 fd 必须用 WireGuard utun 遍历法（全 iOS 12~26 统一）
禁止调用任何官方 packetFlow 读写方法
Mihomo 启动必须极速启动（<15 秒）
TUN 读写强制单线程
关闭 Mihomo 非必要模块（减少内存）
禁止日志打印过多（极度耗内存）
启动后立刻调用 completionHandler(.connected)
1. TUN fd 必须在拓展内获取，且只能获取一次
必须在 startTunnel 内调用
必须在 packetFlow 创建完成后
早了没有，晚了丢包
2. 一旦使用 fd 接管，绝对不能再调用官方读写方法
禁止调用：
swift
packetFlow.readPackets()
packetFlow.writePackets(...)
混用 = 立即内核冲突、闪退、断流。
3. 拓展内存硬上限 20~25MB（iOS 16+）
Mihomo 启动瞬间内存就可能超
高流量转发会持续涨内存
超了直接 OOM kill，无日志
4. 拓展启动超时 25~30 秒
Mihomo 配置大、规则多 → 启动超时 → 系统强制断开。
5. TUN 设备单线程读写，不支持多队列 / 并发
iOS utun 不支持多队列多线程 read/write → 数据错乱、内核报错。
6. 拓展不能做耗时异步操作
不能请求 HTTP、不能解析大量域名、不能加载大规则库否则直接启动超时。
7. 锁屏 / 后台 / 无流量 拓展会被系统休眠
没有任何合法保活手段，小火箭也不行。
8. iOS 16+ 必须使用 WireGuard 遍历 fd
KVC 已死，这是唯一稳定方式。
9. 拓展进程崩溃不会带崩主 App
但会导致 VPN 断开，必须做自动重启机制。
