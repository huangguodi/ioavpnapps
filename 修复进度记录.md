二、已确认结论
- [x] 当前 `mihomo-clash/ios/mihomo` 的 iOS 实现不是 `tun.file-descriptor` 直连模式
- [x] 当前 iOS 实现依赖 Swift 层通过 `packetFlow.readPackets` / `packetFlow.writePackets` 做数据中转
- [x] Go/mihomo 在这里主要承担流量处理，不直接读写 iOS `utun fd`
- [x] `mobile.go` 会对 iOS 配置做收敛处理，`tun.file-descriptor` 和注入的 `dns-hijack` 不能作为最终有效依据
- [x] 仅依靠调低 MTU、关闭 IPv6 默认路由、调整 TUN 地址，仍未解决整机断网

5. 最新真机日志结论（`2026-04-01`）
- [x] 日志显示 `start requested -> network settings applied -> packet flow bridge installed -> core started` 启动链路完整
- [x] 日志显示 `read packets total=600`，说明 `packetFlow.readPackets` 在持续从系统 TUN 侧收到流量
- [x] 日志显示 `feed packet success total=600`，且没有出现 `feed packet failed`，说明 Swift -> `MobileFeedPacketBytes` 入队链路正常
- [x] 当前日志中完全没有 `write packet` 记录，说明 `bridge.writePacket -> packetFlow.writePackets` 这一侧尚未产生任何可回写数据
- [x] 结合现象，高度怀疑是 iOS 14+ Network Extension 沙盒权限坑：即没有显式初始化 `NWPathMonitor`，导致 NE 进程被系统剥夺外网访问权限
- [x] 这会导致：mihomo 收到系统进来的包 -> 尝试通过 TCP/UDP 出站连接代理节点或解析外部 DNS -> 被 iOS 系统直接拦截 -> 没有任何回包 -> 整机无网

6. Network Extension 权限激活修复
- [x] 在 `PacketTunnelProvider.swift` 的 `init()` 中增加 `let _ = NWPathMonitor()` 强制触发网络权限申请
- 结果：`2026-04-01` 真机反馈依然“一模一样，只进不出，整机无网”。这说明**沙盒网络权限坑并不是当前唯一的拦截点**，包确实在内核处理的更深处丢失了。

五、当前判断
- 现象依旧：`read packets` 持续，`feed packet` 成功，但 `write packet` 始终为 0
- 既然 `NWPathMonitor` 未能解决，意味着：
  - 可能是 TUN 桥接层（`MobileFeedPacketBytes`）内部虽然成功将包投递到了 Go channel，但 `sing-tun` 的 `packetFlowTun.Read` 没有被核心的 `tunStack.Start()` 消费。
  - 或者是 `gvisor` 栈虽然读到了包，但 TCP/UDP 连接回调（`NewConnection` / `NewPacketConnection`）没有触发。
  - 又或者是流量成功进入核心 `Tunnel.HandleTCPConn`，但出站解析 DNS 时被拦截或超时，导致连接静默断开。
- 下一步必须通过日志“解剖” `mihomo` 核心的接收链路，不能再在外部瞎猜。
